## 🎯 核心摘要

Claude Code团队对Agent循环（loop）给出了官方定义：loop是Agent重复执行工作周期，直到满足特定停止条件。根据触发方式、停止条件、底层工具和适用任务类型，循环被分为四大类：轮次循环、目标循环、时间循环和主动循环。文章还提供了代码质量保证和Token消耗控制的实用策略，强调用系统设计替代手动提示词是Agent工程化的核心方向。

## 📊 核心观点

### 1. Loop 的官方定义与四类循环

**背景/现状**：
- 技术圈对"loop"有各种不同解释，概念混乱
- Claude Code团队给出了明确的标准化定义

**核心论述**：
- **Loop 定义**：Agent重复执行工作周期，直到满足特定停止条件
- **分类维度**：按触发方式、停止条件、使用的底层工具、适合的任务类型分四类

### 2. 轮次循环（Turn-based loops）

**背景/现状**：
- 最基础的交互模式，每个用户提示词启动一个手动循环

**核心论述**：
- **触发**：用户提示词
- **停止**：Claude判断已完成或需要额外上下文
- **适用**：不属于常规流程的短任务
- **关键改进**：将手动验证步骤编写为SKILL.md技能文件，包含具体的验证工具和可量化的检查标准
- **示例**：`verify-frontend-change`技能——启动开发服务器、交互验证、检查控制台、运行性能审计，任何步骤失败都从第1步重跑

### 3. 目标循环（Goal-based loop - /goal）

**背景/现状**：
- 复杂任务单次沟通不够，Agent在能迭代时表现更好
- Agent自行判断"够了"往往提前结束，达不到最优

**核心论述**：
- **触发**：实时手动提示词
- **停止**：达成目标或达到最大轮次
- **适用**：有可验证退出标准的任务
- **核心价值**：确定性标准（通过测试数量、分数阈值）比模糊标准有效得多
- **示例**：`/goal 将主页Lighthouse分数提升到90分或以上，尝试5次后停止`

### 4. 时间循环（Time-based loop - /loop 和 /schedule）

**背景/现状**：
- 有些Agent工作是定期重复的，任务不变只有输入变化
- 有些依赖外部系统变化，需要定期检查并响应

**核心论述**：
- **触发**：指定时间间隔
- **停止**：手动取消或工作完成（PR合并、队列清空）
- **适用**：定期重复工作或与外部系统交互
- `/loop` 在本地运行，关闭电脑即停止
- `/schedule` 将循环转移到云端，持续运行

### 5. 主动循环（Proactive loops）

**背景/现状**：
- 周期性且定义明确的流水线工作，无需人类实时参与

**核心论述**：
- **触发**：事件或计划任务
- **停止**：每个任务达成目标退出，例行程序持续运行直到关闭
- **适用**：Bug报告处理、问题分类、迁移、依赖升级
- **组合策略**：将例行程序路由到更小更快的模型，需要判断时用能力最强的模型

### 6. 四类循环的组合使用

**核心论述**：
- 主动循环是最高阶模式，可组合所有基础组件：
  - `/schedule`：定期检查新报告
  - `/goal`：定义完成标准和验证方法
  - 动态工作流：协调多Agent并行探索方案
  - 自动模式：无需停止请求许可
- **示例**：每小时检查Bug报告→分类→三种方案并行修复→对抗性审查→完成

## 🔑 关键洞察

### 1. 用系统设计替代手动提示词是核心范式

**分析**：
- 整篇文章的底层逻辑是：不要再手写越来越长的提示词，而是设计循环系统来驱动Agent
- SKILL.md、/goal、/schedule、动态工作流都是"系统"而非"提示"
- 这标志着Prompt Engineering从"写好一句话"进化到"设计一套系统"

### 2. 验证能力的可量化是关键杠杆

**分析**：
- 轮次循环中，验证步骤从"人类看看"进化到"可执行、可量化的检查脚本"
- 目标循环中，确定性标准（分数、测试数）远优于模糊标准
- 本质是：你能多精确地定义"完成"，Agent就能多可靠地交付

### 3. 模型路由是多Agent系统的成本控制核心

**分析**：
- 主动循环中明确建议：例行任务用小模型，判断任务用大模型
- 这与云原生架构中"合适的工具做合适的事"思维一致
- 多Agent系统如果不做模型路由，成本会指数级增长

## 🚧 不足与局限

### 1. Claude Code 专属方案
- 四类循环和 /goal、/loop、/schedule 等指令都是 Claude Code 的特定功能
- 其他 Agent 框架（LangChain、AutoGen、CrewAI等）的实现方式可能完全不同
- 概念可迁移，但具体工具不可直接套用

### 2. 缺少失败案例
- 文章偏向"理想路径"描述，缺少循环失控、Token爆炸、Agent陷入死循环等真实失败场景
- 对于刚接触循环机制的人来说，知道"会出什么问题"比知道"应该怎么做"更重要

### 3. 质量保证部分过于简略
- "保持代码库干净""给Agent验证方法"等建议正确但不够具体
- 缺少在大型代码库中实际应用循环的具体策略和踩坑经验

## 🔮 延伸思考

### 方向1：循环机制的跨框架标准化
- Claude Code 的四类循环是否可以抽象为Agent循环的通用模型？
- 未来可能出现类似 Kubernetes CRI 的 Agent 循环接口标准

### 方向2：循环经济学
- 不同类型循环的Token消耗模型差异巨大
- 可能需要专门的"循环成本估算器"来帮助决策用哪种循环

### 方向3：从循环到自治
- 主动循环已经是高度自治的模式，下一步可能是Agent自主决定使用哪种循环
- 这需要Agent具备"元认知"能力——知道自己需要什么级别的自主性

## 💡 实践启示

### 1. 从审视现有工作开始，而非从选循环开始

**要点**：
- 找到当前瓶颈环节，思考哪部分可以交出去
- 能写出验证检查步骤？→ 轮次循环+SKILL.md
- 目标足够明确？→ 目标循环 /goal
- 工作定期出现？→ 时间循环 /loop 或 /schedule

### 2. 把验证步骤产品化

**要点**：
- 将你和团队认可的验证标准编写为SKILL.md技能文件
- 越具体、越量化，Agent自我验证越可靠
- 包括实际的工具和连接器，让Agent能"看到"结果而不只是"猜测"

### 3. 先小规模试点再扩展

**要点**：
- 动态工作流可能生成数百个Agent，在大规模运行前必须先试点
- 用 /usage、/goal、/workflows 命令监控Token消耗
- 随时可以停止Agent，保持控制权

### 4. 系统性改进而非单点修复

**要点**：
- 当单个结果未达标准时，不要只修复那个具体问题
- 将问题编写到系统中，改善未来所有迭代的表现
- 这种"从案例到系统"的思维是Agent工程化的关键

## 📝 关键金句

> "Loop就是Agent重复执行工作周期，直到满足特定的停止条件。"

> "检查越具象和量化，Claude就越容易进行自我验证。"

> "当单个结果未达到标准时，不要仅仅停留在修复该具体问题上，尝试将其编写到系统中，以改善未来所有迭代的表现。"
